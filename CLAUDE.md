## 项目性质

本项目为**研究型项目**，不是工程型项目。  
所有开发行为必须服务于研究目标，而非工程完整性或商业产品化。

---

# 开发规则

## 研究型代码开发规范

### MVP开发准则
- 不允许随意添加测试代码
- 不允许使用默认值
- 避免通过默认值掩盖错误，从而造成逻辑错误

### 类型注解规范
- 所有方法签名必须包含完整类型注解
- 复杂类型必须使用 `Union`、`Dict` 等明确表达

### 文档规范
- 所有模块、类、方法必须包含 **中文 docstring**
- docstring必须说明：
  - 功能
  - 参数
  - 返回值
  - 关键实现细节

### 代码组织模式
复杂逻辑必须使用**阶段化注释**组织，例如：
第一阶段：数据预处理
第二阶段：特征构建
第三阶段：模型计算
以提高代码可读性和研究复现性。

### 依赖管理
导入顺序必须严格遵循：

1. Python标准库
2. 第三方库
3. 项目内部模块

每个模块只导入**必要依赖**。

### 命名规范
- 类名：`PascalCase`
- 变量名：使用**语义清晰的描述性命名**
- 未使用但必须保留的变量：使用 `_` 前缀

### 接口设计原则
函数返回值必须包含完整诊断信息，例如：

- 输出结果
- 损失值
- 指标
- 统计信息

必要时使用 **条件标志控制返回内容**。

### 错误处理与验证
必须进行运行时验证：

- 张量维度一致性
- 设备一致性（CPU/GPU）
- 输入数据合法性

所有代码文件必须保证：

- 能通过语法检查
- 不存在隐藏错误

# Git 版本管理规则
本项目使用 **Git 进行版本管理**。
因此：
- 所有修改必须 **直接在原文件中进行**
- **禁止创建迭代版本文件**（如 `xxx_v2.py`）
- **不需要保持向后兼容**
---
# 提交规范
每一次提交必须包含：
- 清晰
- 简洁
- 可读
的提交信息。

## Conventional Commits
必须使用 **Conventional Commits 规范**：
feat: 增加新功能
fix: 修复现有问题
docs: 文档相关修改
refactor: 代码重构
---
# Git 工作流规范
- **版本策略**: 不考虑向后兼容，直接修改原文件。代码简洁性优先。
- **提交规范**:
  - 格式: Conventional Commits (`feat:`, `fix:`, `docs:`, `refactor:`)。
  - 信息: 清晰、简洁、可读。
  - **防御性提示**:
    > [!CAUTION]
    > **严禁** 在 commit message 中添加任何 AI 标识 (如 `Co-Authored-By: Claude`, `🤖 Generated with...`)。
    > 所有提交必须仅显示人类开发者为作者。

commit message 必须：
- 简洁
- 专业
- 不包含任何 AI 工具标记
